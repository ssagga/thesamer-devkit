# v0.2 — Real-world hardening backlog

**Source:** First real-world deployment of the kit — installed into and used on a live project
(`thesamer.com-v5`: Next.js 15 + React 19, `better-sqlite3` CMS, custom auth, Docker→GHCR→Portainer
deploy with `main` = CI-only and `production` = deploy). The kit worked end-to-end (shipped a
feature, then diagnosed + fixed a production bug through the lifecycle), but dogfooding surfaced the
gaps and missing workflow steps below.

**How to work this backlog:** each numbered item is a unit of work — branch per item, spec the
substantial ones, ADR when there's a real trade-off, keep `CLAUDE.md`/`docs/roadmap.md` current,
and re-run `bin/devkit-init.sh` against a throwaway repo plus the new fixtures to confirm the
scaffold still lands. Suggested order: **A → B → E quick wins → the cross-cutting preferences
mechanism → C and D** (the substantial workflow features).

## Keep — validated, do NOT change
- The feature lifecycle (spec → sign-off → branch → build → verify → PR → merge → deliberate promote).
- Sub-agent delegation (Explorer kept context lean and produced a brief good enough to diagnose a
  subtle prod bug).
- Model-tier routing (Sonnet implements, Opus reviews/diagnoses).
- The two-gate model (merge ≠ deploy).
- "Docs currency = done" — the ADR + `CLAUDE.md` gotcha fenced the prod bug off from regressing;
  this was the single highest-value behavior observed.

## Cross-cutting — "involvement preferences" (build first; items 6, 7 depend on it)
Introduce ONE preferences mechanism the installed system reads from (e.g. a `## Working preferences`
block in `CLAUDE.md`, or `.claude/devkit.json`), captured by `devkit-init` by asking the user once.
At minimum: **plan gate** (`always`/`substantial-only`/`never`), **preview gate**
(`always`/`visual-only`/`never`), and confirm **merge/promote stay human** (never auto). Every gate
below reads this instead of hardcoding behavior.

---

## A — Installer correctness & safety (highest priority)

1. **Recursive + dependency-aware store detection** — `bin/devkit-init.sh` (`detect_stack`).
   It only globs the repo root for `*.sqlite`/`*.db` and missed `data/thesamer.db`, so it reported
   "store: none" and advised *deleting the Data-safety section* for a DB-backed CMS. Search common
   subdirs (`data/`, `db/`, `prisma/`, `.data/`) and infer the store from `package.json` deps
   (`better-sqlite3`, `drizzle-orm`, `@prisma/client`, `pg`, `mysql2`, `mongodb`). On uncertainty,
   **keep** the Data-safety section + emit a TODO — never advise deleting it on a false negative.
   Add a nested-SQLite fixture to the verify plan.

2. **Never let the installer's own files be gitignored** — `bin/devkit-init.sh` post-install check.
   After scaffolding, `git check-ignore` the files it wrote (esp. `.claude/skills/pre-pr-review/`).
   If ignored (common: a blanket `/.claude/skills/`), auto-add a narrow negation
   (`!/.claude/skills/pre-pr-review/`) or warn loudly. The review gate must survive a fresh clone.

3. **Generate a runnable CI workflow** — `templates/.github/workflows/ci.yml` + fill logic.
   Emit an actually-runnable job for the detected stack (filled `setup-node` + package-manager
   setup), not an all-commented template. When `package.json` has a `packageManager` field, do
   **not** also pin the version in `pnpm/action-setup` — that combo fails with
   `ERR_PNPM_BAD_PM_VERSION`. Add a `packageManager`-pinned fixture; confirm the workflow is valid.

## B — Brief & branch-model fidelity

4. **Infer the live/deploy branch from existing workflows** — `detect_stack` + `CLAUDE.md` fill.
   Parse `.github/workflows/*.yml` for `on.push.branches` and `if: github.ref == 'refs/heads/<x>'`
   to distinguish integration vs deploy branches, instead of defaulting live=integration. Real
   project: `main` (CI) + `production` (deploy).

5. **Add a "build-time vs runtime data" hazard to the template** — `CLAUDE.template.md`
   (Data-safety/gotchas). Generalize the production bug: *if the build can't see the production
   data store, do not statically generate pages that read it — they bake a build-time snapshot and
   silently serve stale/seed data after every deploy.* Recommend dynamic rendering or runtime
   caching with on-demand invalidation for store-backed pages.

## C — Workflow & human-in-the-loop (new)

6. **Plan-presentation gate** — new `templates/.claude/agents/plan-presenter.md` + update the SPEC
   step in `CLAUDE.template.md`. The kit has `explorer` and `planner`; what's missing is a final
   **consolidation that presents exactly what will be done before a line of code is touched**, then
   waits for go/no-go. Synthesize Explorer findings + Planner approach + draft spec into one
   concrete, approval-ready summary: scope, files to create/change, approach, data/migration impact,
   verify plan. Fires per the **plan gate** preference; on Claude Code maps to plan mode. When off,
   planning still happens — it just doesn't stop for sign-off.

7. **Optional local-preview gate before verify** — new `templates/.claude/skills/preview/` + insert
   a `PREVIEW (optional)` step between BUILD and VERIFY in the `CLAUDE.template.md` lifecycle. After
   implementation but **before** the agent tests/audits, offer the human a chance to look at the
   change running locally — *especially for visual/UI changes* — instead of running straight to the
   end. Detect app type and launch the right preview (web → dev server + URL/screenshot; mobile →
   simulator; CLI/TUI → run). Fires per the **preview gate** preference. Composes with item 12 —
   build the run-&-observe capability once and reuse it here.

## D — Staying current with the kit (new)

8. **Devkit update-awareness** so installed projects don't drift from kit improvements.
   (a) At install, record provenance — devkit source repo + ref/commit + install date (e.g.
   `.claude/devkit.json`). (b) Ship a `/devkit-update` skill (and/or `devkit-init.sh
   --check-updates`) that compares the recorded ref against the devkit's latest release, summarizes
   what changed, and **applies only kit-owned files** (agent defs, skills, CI template, methodology)
   while **never clobbering project-authored content** (the filled `CLAUDE.md`, `docs/`) — surface a
   diff for anything ambiguous and let the human accept. (c) Maintain a `CHANGELOG.md` and tag
   releases so there's something to diff against. Consider prompting an update check during ORIENT.
   Substantial — write a spec + an ADR on the update/merge strategy (how kit-owned vs project-owned
   files are separated so updates are safe and idempotent).

## E — Enforcement honesty & ergonomics

9. **Be honest about enforcement** — `bin/devkit-init.sh` + README/`CLAUDE.template.md`. Branch
   protection is unavailable on private free GitHub repos (returns `403 — Upgrade to Pro or make
   public`). Detect visibility/plan, set protection when possible, else print the exact options
   (Pro / public / convention-only) and offer a local `pre-push` hook as a convention-only backstop.
   Document that "no unreviewed change reaches live" is *enforced* only with protection, *convention*
   otherwise.

10. **Cut status-flip ceremony** — `CLAUDE.template.md` DoD + PR template. State that roadmap status
    updates ride in the **same PR** that ships the work (mark it shipped there, since merging ships
    it) — not as separate status-only PRs.

11. **Add a read-only `/status` digest skill** — `templates/.claude/skills/status/`: current branch,
    roadmap now/next, open PRs + CI state, any data-safety flags — a projection of the source of
    truth (no stored state) to re-orient between sessions.

12. **Smooth "run & observe"** — docs + the shared capability behind item 7. Document fallbacks when
    `pnpm`/node tooling isn't on PATH (use `node_modules/.bin`), and a per-app-type way to launch +
    screenshot the app.
